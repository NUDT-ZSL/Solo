# NebulaWeaver 连线网络生成性能评估

## 一、问题背景

连线网络生成需要计算所有粒子对之间的距离，筛选出距离小于阈值的粒子对并绘制连线。随着粒子数量增加，暴力遍历的计算量呈平方级增长，严重影响用户体验。

**需求指标**：
- 800 个粒子构建网络耗时 < 500ms
- 3000 个粒子构建网络耗时 < 2s（推论）

## 二、方案对比

### 方案 A：暴力双重遍历（Brute Force O(n²)）

**原理**：遍历所有粒子对 (i, j) 其中 i < j，计算欧氏距离判断是否连线。

```
for i in [0, n):
    for j in [i+1, n):
        dist = distance(particles[i], particles[j])
        if dist < threshold:
            add_edge(i, j)
```

**时间复杂度**：O(n²)
**空间复杂度**：O(n) （存储位置）

| 粒子数 | 距离计算次数 | 预估耗时 (js) |
|--------|-------------|---------------|
| 500    | 124,750     | ~10ms         |
| 800    | 319,600     | ~25ms         |
| 1000   | 499,500     | ~40ms         |
| 2000   | 1,999,000   | ~160ms        |
| 3000   | 4,498,500   | ~360ms        |

**优点**：
- 实现简单，代码量少
- 无额外内存开销
- 对于 n < 1000，性能完全足够

**缺点**：
- 计算量随粒子数平方增长
- 粒子数超过 3000 时性能急剧下降
- 做了大量无意义的远距离比较

---

### 方案 B：均匀网格空间索引（Uniform Grid / Spatial Hashing）

**原理**：
1. 将三维空间划分为边长等于距离阈值的立方体网格
2. 每个粒子分配到所在网格单元
3. 查询时只检查目标粒子所在单元及其 26 个相邻单元中的粒子

```
┌─────┬─────┬─────┐
│     │     │     │
│  ●  │     │     │
│     │     │     │
├─────┼─────┼─────┤
│     │     │     │
│     │  ●  │  ●  │  ← 只需检查相邻9个(2D)/27个(3D)单元
│     │     │     │
├─────┼─────┼─────┤
│     │     │     │
│     │     │     │
│     │     │     │
└─────┴─────┴─────┘
```

**时间复杂度**：
- 建网格：O(n)
- 查询：O(n × k)，其中 k 是每个网格平均粒子数
- 总体：O(n)（当粒子分布均匀时）

**空间复杂度**：O(n + m)，m 为网格单元数

| 粒子数 | 网格单元数 | 平均每格粒子 | 距离计算次数 | 预估耗时 |
|--------|-----------|-------------|-------------|---------|
| 500    | ~220      | ~2.3        | ~1,500      | ~3ms    |
| 800    | ~350      | ~2.3        | ~2,400      | ~5ms    |
| 1000   | ~440      | ~2.3        | ~3,000      | ~6ms    |
| 2000   | ~870      | ~2.3        | ~6,000      | ~12ms   |
| 3000   | ~1300     | ~2.3        | ~9,000      | ~18ms   |

**优点**：
- 性能提升巨大（800 粒子时约 50 倍加速）
- 实现相对简单
- 内存开销可控
- 查询是精确的，无遗漏

**缺点**：
- 粒子分布极不均匀时性能退化（但球壳分布较均匀）
- 网格大小需要根据距离阈值调整
- 三维情况下需要查询 27 个相邻单元

---

### 方案 C：八叉树（Octree）

**原理**：
1. 递归将空间八等份，每个节点存储其范围内的粒子
2. 当节点内粒子数超过阈值时继续细分
3. 查询时只遍历与查询球相交的节点

```
         ┌───────────┐
         │     │     │
         │  1  │  2  │
         │─────┼─────│  ← 递归八叉划分
         │  3  │  4  │
         │     │     │
         └───────────┘
```

**时间复杂度**：
- 建树：O(n log n)
- 查询：O(k log n)，k 为邻域粒子数
- 总体：O(n log n)

**空间复杂度**：O(n)

**优点**：
- 对非均匀分布适应性强
- 支持动态插入/删除
- 粒子数极大时优势明显
- 空间划分更智能

**缺点**：
- 实现复杂，代码量大
- 常数因子较大，中小规模下可能不如网格快
- 需要处理节点分裂/合并逻辑
- 调试困难

---

## 三、方案选型：均匀网格空间索引

### 选择理由

1. **场景匹配**：粒子均匀分布在球壳内，密度相对均匀，网格索引效率最高
2. **性能充足**：800 粒子下仅需 ~5ms，远优于 500ms 需求
3. **实现简单**：约 60 行代码，易于维护和验证
4. **内存可控**：每个粒子只存一份索引引用
5. **精确结果**：与暴力算法结果完全一致，无精度损失

### 与八叉树的对比权衡

| 维度 | 均匀网格 | 八叉树 | 结论 |
|------|---------|--------|------|
| 实现复杂度 | 低（~60行） | 高（~200行） | 网格胜 |
| 800粒子性能 | ~5ms | ~15ms | 网格胜 |
| 3000粒子性能 | ~18ms | ~25ms | 网格胜 |
| 10000+粒子 | ~60ms | ~50ms | 八叉树胜 |
| 非均匀分布 | 退化 | 稳定 | 八叉树胜 |
| 动态粒子 | 重建O(n) | 插入O(log n) | 八叉树胜 |
| 调试难度 | 容易 | 困难 | 网格胜 |

**结论**：在当前项目场景（粒子数 < 3000，静态分布，球壳均匀）下，均匀网格是最优选择。

## 五、实现要点

### SpatialGrid 类设计

```
class SpatialGrid {
  - cellSize: number           // 网格边长 = 距离阈值
  - grid: Map<string, number[]> // 网格哈希表
  - invCellSize: number        // 1/cellSize，避免除法

  + build(positions, count)    // 构建网格索引
  + query(x, y, z)             // 查询邻域候选粒子
  + clear()                    // 清空网格
}
```

### 优化技巧

1. **使用整数键**：`Math.floor(x * invCellSize)` 比除法快
2. **字符串键哈希**：`"cx,cy,cz"` 作为 Map 键，简单高效
3. **查询 27 个邻居**：三维空间中目标格 + 26 个相邻格
4. **避免重复计算**：只处理 j > i 的粒子对
5. **距离平方比较**：用 distSq 替代 dist，省去开方
6. **单次构建多次使用**：粒子静态时只需构建一次网格

### 正确性保证

- 网格边长 = 距离阈值，保证相邻粒子一定在相同或相邻网格中
- 不会遗漏任何距离 < 阈值的粒子对
- 结果与暴力算法完全等价

## 六、未来优化方向

如果未来粒子数增加到 10000+ 或需要动态粒子，可以考虑：

1. **升级为八叉树**：更好的非均匀分布适应性
2. **BVH（包围体积层次）**：适合光线投射类查询
3. **k-d tree**：适合最近邻查询
4. **GPU 加速**：使用 compute shader 并行计算
5. **增量更新**：粒子移动时只更新相关网格单元
